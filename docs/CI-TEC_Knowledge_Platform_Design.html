CSP·MSP·레거시 온프레미스 기술지원 전문 부서를 위한 시스템. 틀린 답 한 건이 장애 확대로 이어질 수 있으므로, 화려한 생성보다 정확한 검색·근거 투명성·근거 부족 시 침묵(abstention)을 최우선으로 한다. Docker 6서비스(+옵션 Keycloak)로 가볍게 운영하고, 승인된 인사이트만 지식으로 환류한다. v2.4는 설계뿐 아니라 2026-07 As-Built(하이브리드 검색·플래너·MCP·폐쇄망 배포)를 문서화한다.

이상적 시스템의 본질 (통찰)
기술지원 조직에서 지식시스템의 가치는 “똑똑한 문장”이 아니라 (1) 맞는 사례를 빠르게 표면화하고 (2) 그 사례가 현재 환경(CSP/MSP/온프레, 버전, 구성)에 적용 가능한지를 가르며 (3) 틀릴 가능성을 숨기지 않는 것이다. 이상적 설계 = 검색 엔진의 정밀도 + 증거 등급 체계 + 환경 축(environment axis) + 사람 검수 환류.

1 부서 맥락과 문제 정의

CI-TEC = CSP · MSP · Legacy On-Prem 종합 기술지원 전문가 집단

1.1 업무 특성 (설계를 지배하는 사실)

의사결정의 대가

잘못된 커널 파라미터·네트워크 MTU·DB failover 판단은 고객 장애로 직결된다. 따라서 시스템은 확신을 연출하면 안 되고, 근거와 한계를 드러내야 한다.

환경 이질성

동일 증상(“IO 편차”, “heartbeat loss”)도 CSP 가상화 / MSP 운영대행 / 레거시 베어메탈에서 원인·조치가 다르다. 검색·답변은 반드시 환경 축을 인식해야 한다.

지식 형태

정형 runbook보다 사례(티켓) + 원리(테크리포) + 점검(PISA) + 심층분석(tuning_ai)의 조합이 실무 답이다. 단일 문서 FAQ 봇이 아님.

사용자

초급~시니어 혼재 50–100명. 시니어는 “키워드·티켓키 정확 검색”, 주니어는 “증상 서술 QA”를 더 쓴다. 두 모드 모두 1급 시민.

1.2 As-Is Pain

  • Jira 지원이력 · Confluence 테크리포 · PISA 점검 · AI 분석물이 raw/에 쌓이지만 교차 검색·환경 필터·신뢰 등급이 없음.
  • “예전에 풀었던 그 장애”를 찾기 위해 시스템·사람 의존 탐색 → 응답 시간·품질 편차.
  • 인사이트가 개인 메모에 머물고 조직 지식으로 환류되지 않음.
  • 인터넷 없는 폐쇄망 → 외부 SaaS RAG 불가.

1.3 To-Be

엔지니어가 증상·환경·제품을 입력하면, 검증된 근거 문서만 순위화하여 보여주고, 필요 시 GLM이 근거 범위 안에서 해석·비교 인사이트를 생성하며, 시니어가 승인한 패턴만 다시 검색 가능한 지식으로 축적되는 플랫폼.

2 목표 · 비목표 · 품질 SLA

“모든 검색결과가 정확하고 신뢰도 높아야 한다”를 측정 가능한 계약으로 번역

Goals

  1. 고정밀 통합 검색 — 티켓키·파라미터·제품명 정확 매칭 + 의미 검색.
  2. 환경 인식 필터 — CSP / MSP / On-Prem · 제품 영역 · 지원유형.
  3. 근거 기반 답변 — citation 없는 단정 금지, 기권 가능.
  4. 신뢰 신호 — 단일 %가 아닌 다차원 Trust UI (§5).
  5. 경량 Docker 운영 — 5 컨테이너 전후, 부서 단독 호스팅 가능.
  6. 지식 환류 — 승인된 인사이트만 색인.
  7. 폐쇄망 — 외부 egress 0, 사내 GLM 5.2 130k.

Non-Goals

  1. 자동 원격 조치 / 스크립트 실행 오케스트레이션.
  2. 전사 ACL 실시간 미러 (초기 부서 단일 테넌트).
  3. Kafka·ES 클러스터·GPU 임베딩 상시 전용 노드 필수화.
  4. LLM 파인튜닝 파이프라인 (안정화 이후).
  5. 이미지 OCR·로그 스트림 실시간 분석 플랫폼 대체.

품질 SLA (Precision-first Contract)

지표목표의미
Precision@3 (검색) ≥ 0.90 상위 3건 중 관련 문서 비율. “정확한 결과”의 1차 정의.
nDCG@5 ≥ 0.85 순위 품질. gold set 100질의 기준.
Abstention precision ≥ 0.95 “근거 부족”이라고 한 경우 실제로 근거가 없었던 비율.
Answer groundedness ≥ 0.95 문장 생성 문장이 citation 청크에 매핑되는 비율.
Critical false advice rate ≈ 0 (샘플 감사) 조치 명령을 틀리게 단정한 비율. 발생 시 즉시 롤백 검토.
검색 p95 ≤ 700ms 리랭크 포함 시 ≤ 2.0s (100명 피크 가정).
Fast QA p95 / TTFT ≤ 12s / ≤ 2.5s 기본 프로필(근거 8–20k 토큰).
가용성 업무시간 99% 부서 내부 도구 수준. 다중 DC 불필요.
List Completeness (exhaustive 질의) ≥ 0.98 “모두 찾아줘” 유형에서 필터 집합 대비 누락률. top-k RAG 금지 지표 (§3D.7)
!
“모든 결과가 정확”의 공학적 해석
정보검색에서 recall 100% + precision 100%는 불가능에 가깝다. 본 설계는 사용자에게 보이는 상위 결과의 precision과 생성 답변의 groundedness를 극단적으로 올리고, 확신이 없으면 결과를 줄이거나 기권한다. “많이 보여주기”가 아니라 “틀리지 않게 보여주기”가 계약이다.

3B.6 GRO offload — 실측 증거

  • CITECTS-2171 산호세 SRA 파일 전송: Broadcom bnxt_en + VXLAN 환경에서 GRO HW offload가 Pod throughput 저하. 조치: ethtool -K <dev> rx-gro-hw off, 펌웨어 업그레이드 권고.
  • CITECTS-2557 네이버웍스 VM timeout: HPE 호스트 rx-gro-hw=on 성능 저하. 동일 패턴 인용(2171). 조치: off + Live migration.
  • tech_repo는 TCP offload 일반(ethtool -k/-K … gro on) 튜닝 문서 존재 — 장애 사례와 구분 필요.

→ “모든 GRO 이슈 종합” 질의는 엄격 토큰 매칭 + 동의어(rx-gro-hw, hardware GRO, generic receive offload) 와 교차 인용 그래프로 답 가능. 문서 수는 적으나 품질·재현성이 높아 Trust HIGH 가능.

3G 그룹장 유사장애 질의 — 1급 유스케이스

“(지금 장애가 났다) 예전에 비슷한 게 있었나? 뭐였지? 어떻게 풀었지? 이번에도 그 방법이 먹히나?” — 부서장/그룹장의 mid-incident 의사결정 질문. 단순 검색이 아니라 유사도 + 적용 가능성 판정이다.

3G.1 질문 분해 (반드시 4슬롯)

Slot 1 · Similarity

이전에 유사 장애가 있었는가?

Top-K 유사 사례 카드 (있음/약함/없음). 없음이면 기권+추가 수집 질문.

Slot 2 · What

어떤 장애였는가?

증상 · 영향 범위 · 환경(CSP/온프레) · 제품 · 기간. Issue Frame 요약.

Slot 3 · How fixed

어떻게 해결했는가?

조치 · workaround · 명령 · 담당/벤더. 원인+조치 동시 있는 사례 가중 (코퍼스 ~516건).

Slot 4 · Applicability ★

이번에도 그 해결이 도움이 되는가?

현재 장애 컨텍스트와 과거 사례의 차이점을 비교. 여기가 그룹장 질문의 핵심이며 일반 RAG가 가장 쉽게 거짓 확신을 내는 지점.

★
통찰: “유사”와 “적용 가능”은 다른 축
증상 텍스트가 비슷해도 환경이 다르면 해결책이 독이 될 수 있다 (예: 과거 GRO offload off 조치 ≠ 현재 물리 NIC/드라이버가 다른 구성). 시스템은 유사 사례 검색(Retrieval)과 적용 가능성 판정(Applicability)을 분리하고, 후자는 환경·제품·버전·구성 슬롯 일치도로 Trust를 매긴다. 슬롯이 비어 있으면 “도움이 된다”고 단정하지 말고 확인 질문을 한다.

4 설계 원칙 — Trust-First Knowledge System

이상적 지식검색의 7원칙

P1

Precision over Coverage

상위 결과 오탐이 한 건이라도 치명적. 애매하면 결과 수를 줄이고 기권한다.

P2

Evidence over Eloquence

유창한 문장보다 출처·재현 가능한 명령·티켓 링크. LLM은 해석자, 증거가 주연.

P3

Environment is a First-class Axis

CSP / MSP / On-Prem · 제품 스택 · 버전을 필터·랭킹·답변 경고에 넣는다.

P4

Hybrid Retrieval is Non-negotiable

기술지원 질의는 ID·파라미터·에러코드가 핵심. 벡터 alone 금지.

P5

Calibrated Silence

모르는 것을 “그럴듯하게” 말하지 않음. 기권 + 추가 수집 질문 + 검색 키워드 제안.

P6

Human Gate on Flywheel

생성 지식의 자동 색인은 금지. 시니어 승인 후에만 조직 지식.

P7

Operational Minimalism

50–100명·폐쇄망·부서 운영 → 컴포넌트 최소화. 한 팀이 밤샘 없이 운영 가능해야 이상적.

P8

Measure or Don’t Ship

gold set·주간 회귀 없이 모델/프롬프트 변경 금지. 품질은 느낌으로 관리하지 않는다.

5 신뢰도(Confidence)에 대한 깊은 통찰

“답변에 신뢰도를 포함하는 것이 의미 있는가?” — 결론부터: 의미 있다. 단, 잘못된 형태의 신뢰도는 해롭다.

5.1 왜 단일 “신뢰도 87%”는 위험한가

Anti-pattern

스칼라 확률 연출

  • “답변 신뢰도 87%”처럼 보이는 숫자
  • 모델 logprob나 임의 가중 합산
  • 교정(calibration) 없는 점수
  • 사용자는 87%를 “거의 확실”로 오해 → 검증 생략

기술지원에서 가장 위험한 UX: 틀린 조치 + 높은 %.

Recommended

다차원 Trust Signal

  • 증거 충분성 (근거 수·합의)
  • 증거 등급 (Resolved 티켓 vs draft)
  • 환경 일치 (CSP/온프레 불일치 경고)
  • 검색 합의 (BM25∩Vector)
  • 명시적 기권 / 부분 답변

숫자는 보조, 라벨·근거·경고가 주 인터페이스.

5.2 이상적 Trust 모델 — 세 층

층이름산출사용자에게 보이는 형태
L1 Retrieval Trust 상위 hit 점수 간격, BM25·벡터 교집합 크기, 메타 필터 일치 결과 카드의 “매칭 강함/보통/약함”, 점수 바(상대)
L2 Evidence Grade 소스 타입 × 문서 상태 × 신선도 × 승인 여부 배지: Resolved 사례 공식 문서 점검기준 승인 인사이트 AI초안

5.3 사용자에게 제시할 Trust Banner (권장 UX)

HIGH

근거 충분 · 일치

Resolved 사례 ≥1 + 공식문서/점검 보강. 환경 태그 일치. 교차 검증됨. 조치 전 그래도 환경 확인 문구 고정.

MEDIUM

참고 가능

유사 사례는 있으나 버전/환경 불일치 가능. 차이점 명시. 시니어 확인 권고.

LOW

가설 수준

근거 약함·충돌·미해결 티켓 위주. 단정적 조치 문구 금지. 체크리스트만 제시.

ABSTAIN

근거 부족 · 기권

답변 본문 생성 안 함(또는 “모른다”). 추가 정보 질문 + 추천 검색어 + 관련 문서 링크만.

5.4 점수 산출 (내부용 · 교정 가능)

내부적으로는 0–1 점수를 쓰되, UI 기본은 4단 라벨만 노출. 고급 모드에서만 세부 분해.

retrieval_score = RRF_norm + 0.15 * agreement(BM25, dense) + 0.10 * exact_id_hit
evidence_score  = grade_weight(source, status) * recency_factor * env_match
faithfulness    = cited_sentence_ratio - contradiction_penalty

# 기권 규칙 (예시, 평가셋으로 튜닝)
if retrieval_score < θ_r or evidence_score < θ_e: ABSTAIN
elif faithfulness < θ_f: regenerate once → still low → ABSTAIN / LOW
else map to HIGH / MEDIUM / LOW by calibrated thresholds
★
통찰: 신뢰도는 “모델의 기분”이 아니라 “증거의 구조”
기술지원 전문가에게 설득력 있는 신뢰 신호는 확률 숫자가 아니라 “닫힌 티켓 3건이 같은 원인을 가리키고, 테크리포 파라미터 문서와 일치하며, 당신 환경(온프레 HANA)과 태그가 같다”는 구조적 설명이다. 따라서 시스템은 신뢰도를 계산할 뿐 아니라 설명해야 한다. Explainable confidence > Numeric confidence.

5.5 의미 있는가? — 최종 답

  • Yes — 주니어 과신 방지, 시니어 우선순위 결정, 감사·품질 개선에 필수.
  • But — 단일 % 배지는 No. 4단 라벨 + 근거 설명 + 환경 경고 + 기권이 Yes.
  • Also — 검색 결과 카드에도 동일 언어 사용 (답변만의 장식이 아님).

9 검색·답변 파이프라인 (정확성 중심)

모든 단계에 Quality Gate

9.1 Ingest (정확성의 시작)

  1. 소스 어댑터 → Canonical Document (hash upsert).
  2. Cleaner: Jira 마크업 제거, 빈 본문 drop, 코드/파라미터 보존.
  3. 메타·택소노미 태깅 (규칙 우선).
  4. Evidence grade 부여 (resolved / official / draft…).
  5. 섹션 분할 → contextual header chunk (≤512 tok, overlap 10–15%).
  6. FTS 문서 + embedding 배치. 실패는 DLQ, 부분 성공 표시.

9.2 Search path (고정밀 · As-Built)

  1. 질의 정규화 + 티켓키/파라미터 exact 패턴 추출. 존재 의문형 껍질 제거(「…가 있는가?」).
  2. 사용자 필터 (environment, domain, work_type, source_type).
  3. multi-query: 원문·구문 우선. 단독 고DF 토큰(SCP, Multi-AZ, 연도 alone)은 헤드에서 배제·가중 하향 — 오염 방지.
  4. 병렬: FTS + vector. exact_id 히트는 강제 상위.
  5. RRF 융합 + grade boost + 확장 쿼리 가중 병합.
  6. Quality gate / Trust: 근거 약하면 기권·약함 배너. 모든 hit에 path·body_api·web_url 부착.

9.2b Query Planner 라우트

우선순위: capacity → analytics → time_scoped_list → SI → prevention → exhaustive → checklist → entity_aggregate → hybrid_search.

  • 기간 목록: GET /v1/tickets?relative=
  • 집계: GET /v1/analytics/tickets?group_by=year|issue_type|…
  • 유사장애: POST /v1/similar-incident
  • 통합 NL: POST /v1/query · UI/MCP 동일

9.2c (설계 원문 유지) 게이트 상세

  1. 사용자 필터 (environment, domain, work_type, source, 기간).
  2. 병렬: FTS top 40 + vector top 40. exact_id 히트는 강제 상위.
  3. RRF 융합 + grade boost + env boost.
  4. Quality gate: 상위 스코어 절대/상대 임계 미달 시 결과 축소 또는 empty+가이드.
  5. 다양성: 동일 티켓 중복 청크 병합(parent).
  6. 카드 렌더: 제목, 등급 배지, 환경 태그, 매칭 강함, 스니펫, 링크.

9.3 QA / Insight path

  1. 검색 top N → parent expand → budget pack (Fast 기본).
  2. Trust Engine 사전 판정 → ABSTAIN이면 생성 스킵.
  3. GLM: 근거 내 답변, 문장 단위 citation, 환경 주의 문구.
  4. 사후 faithfulness 체크(규칙/짧은 검증 프롬프트) → 실패 시 재시도 1회 또는 기권.
  5. Trust Banner + 설명형 근거(“왜 HIGH인가”) 표시.

9.4 프롬프트 계약 (요지)

당신은 CI-TEC 기술지원 보조다. 최종 조치 책임은 엔지니어에게 있다.
- 근거 문서에 없는 명령·수치·원인 단정 금지
- 환경(CSP/MSP/온프레) 불일치 시 반드시 경고
- 조치 제안은 "점검 항목" 형태로, 파괴적 명령은 재확인 요청
- 근거 부족 시 한 줄로 기권하고 필요한 추가 정보를 질문
- 각 불릿에 [C#] citation

15 결론 및 권고

  1. 이상적 시스템은 “더 큰 모델”이 아니라 “더 정직한 검색 + 환경 인식 + 기권”이다. CI-TEC처럼 장애 비용이 큰 조직에서는 이것이 제품 차별화의 전부다.
  2. 신뢰도 표시는 필요하다 — 다만 다차원·설명형으로. 단일 87%는 배제. HIGH/MEDIUM/LOW/ABSTAIN + “왜 그런지” + 환경 경고.
  3. Docker는 5컨테이너로 충분하다. 50–100명 규모에서 ES/Qdrant/Kafka는 초기 필수가 아니다. Postgres hybrid가 이상적 균형점.
  4. CSP/MSP/On-Prem 축을 1급 시민으로. 같은 증상 다른 환경 = 다른 지식. 필터·랭킹·Trust에 모두 반영.
  5. 착수 순서: gold set(Q-A/B/C 포함) → compose 골격 → 고정밀 검색(P@3) → 엔티티·checkitem·exhaustive planner → Trust QA → Flywheel. 검색 품질 게이트를 통과하기 전 생성 기능을 키우지 말 것.
  6. 코퍼스 검증 결론: 데이터는 세 예시 질의를 뒷받침하기에 양적으로 충분하거나(모니모·PISA)·질적으로 충분(GRO)하다. 부족한 것은 문서 양이 아니라 엔티티·구조화·집계형 검색 경로이며, §3D로 보완한다.
  7. 100문 gold (v2.2): 키워드 none=0. 완전 답변은 검색/RAG 76% + Planner 9% + Capacity 8% + Analytics 7%. 출시 게이트 = query_catalog_100 평가 통과. 관리자 숫자 질문은 SQL/Rules만 정답.
